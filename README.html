Overview

Argus는 CVE 취약점을 수집·분석하고 탐지 룰을 자동 생성하는 CVE 전용 위협 인텔리전스 파이프라인입니다. GitHub Actions 스케줄러로 주기 실행되며, 결과는 Slack과 GitHub Issue로 알림되고 GitHub Pages 대시보드에 누적됩니다. 개인이 AI 바이브 코딩으로 개발했지만 실제 회사 자산 점검에 활용하는 것을 전제로, 데이터 소스 라이선스와 개인정보 취급까지 설계에 반영했습니다.

왜 만들었나

소규모 조직의 보안 담당자는 대개 겸직이고, 상용 TI(Threat Intelligence) 구독이나 SOC 인력이 없습니다. 그 결과 취약점 공개와 초기 대응 사이의 공백이 위험 구간이 됩니다. Argus는 이 공백을 "공개 → 자동 분석 → 룰 → 알림"으로 메워, 전담 인력 없이도 하루 안에 위험도를 파악하고 탐지 룰을 확보하도록 돕는 것을 목표로 합니다.

핵심 가치

  • 누락 0 설계: "마지막 처리 지점(워터마크)" 이후 전체를 조회하는 방식으로, 스케줄이 수 시간~수일 밀려도 그 사이 CVE를 빠뜨리지 않음
  • 완전 자동화: CVE 공개 → AI 분석 → 탐지 룰 → 담당자 알림까지 사람 개입 없이 동작
  • AI 위협 분석: LLM 기반 근본 원인 분석, MITRE ATT&CK 공격 시나리오, 영향도 평가, 대응 권고 자동 생성
  • 다중 엔진 탐지 룰: Sigma(필수) · Snort/Suricata(네트워크) · YARA(로컬)를 CVSS 공격 벡터에 맞춰 자동 선택 → 보안 장비에 바로 적용
  • 익스플로잇 근거 게이트: KEV·SSVC Active·PoC·EPSS 등 실제 악용 근거가 있는 CVE에만 AI 룰을 생성해 무료 토큰 절약
  • 무료 티어 완주 설계: Gemma/Groq 이중 서비스 · 단일 워커 · 분/일 토큰(TPM/TPD/RPD) 선제 관리로 429 없이 안정 실행
  • 라이선스 준수: 공개 룰 재게시 시 출처·author 보존, PoC 원문 미게시(링크만), 무료 AI에 내부 자산 정보 미전송

Zero-Omission 수집 (워터마크)

보안 모니터링에서 가장 치명적인 결함은 취약점 누락입니다. GitHub Actions 무료 cron은 정시 실행을 보장하지 않아 평균 수 시간, 때로는 하루 이상 지연됩니다. "최근 N시간" 같은 고정 시간창 방식은 실행 간격이 창보다 길어지는 순간 그 사이에 공개된 CVE를 영구히 놓칩니다.

Argus는 시간창 대신 워터마크(마지막 처리 지점) 북마크 방식을 사용합니다. 마지막으로 처리한 커밋 시각을 git에 커밋되는 상태 파일(docs/data/pipeline_state.json)에 기록하고, 다음 실행은 그 지점 이후 전부를 조회합니다. 스케줄이 아무리 밀려도 빈틈이 생기지 않으며, 데이터베이스를 추가로 쓰지 않습니다(불변 원칙: DB 최소화).

메커니즘동작
FIFO 처리커밋 시각 오름차순으로 처리 → 워터마크가 항상 "처리 완료 지점"만 전진
미처리분 보존실패·상한 초과로 이월된 CVE의 최소 커밋 시각 앞까지만 워터마크 전진 → 다음 실행에서 재수집
중복 제거content_hash 비교로 이미 처리한 CVE는 재처리 안 함 (알림 중복 없음)
경계 겹침워터마크에서 5분 겹쳐 조회해 경계 커밋 누락 방지 (중복은 해시가 흡수)
타임아웃 안전워터마크는 실행 끝에 저장 → 30분 타임아웃으로 강제 종료돼도 다음 실행이 그대로 재수집
결과: 실행 주기가 1시간이든 3일이든, 처리에 실패하든 타임아웃으로 죽든 — 공개된 CVE는 언젠가 반드시 처리됩니다. 스케줄 신뢰성에 무관한 누락 0 보장.

AI 룰 생성 게이트 (Exploitation Evidence Gate)

모든 고위험 CVE에 AI 룰을 생성하면 무료 토큰(TPD)이 빠르게 소진됩니다. 실제 악용 근거가 있는 CVE에만 AI 룰을 생성하고, 나머지는 공개 룰 검색만 수행합니다.

Free-Tier Engineering

"100% 무료 운영"은 단순한 비용 절감이 아니라 이 프로젝트의 설계 제약입니다. 무료 티어의 촘촘한 한도 안에서 파이프라인이 중단 없이 완주하도록 여러 전략을 조합했습니다.

이중 AI 서비스로 사용량 분산

번역은 Google Gemma 4, 심층 분석·룰 생성은 Groq Qwen3.6로 분리합니다. 한쪽으로 몰면 일간 토큰이 더 빨리 소진되므로, 서로 다른 무료 티어를 병행해 처리량을 늘립니다.

모델서비스역할일간 한도
gemma-4-31b-itGoogle AI Studio번역·요약RPD 1,500 (요청 수)
qwen/qwen3.6-27bGroq분석·룰 생성TPD 200,000 (토큰)

Groq 무료 티어(TPM 8,000) 대응

Groq는 요청마다 (입력 + max_completion_tokens)를 분당 토큰(TPM)에 선예약합니다. 사고형(thinking) 추론은 단일 요청이 8K TPM을 넘겨 무조건 실패하므로, 무료 티어에서는 reasoning_effort: none + 작은 출력 상한(분석 3,072 / 룰 2,048)으로 구성해 429 폭주를 구조적으로 차단합니다.

선제적 속도·토큰 관리 (Rate Limiter v3.3)

  • TPM 선제 가드: 이번 분 누적 토큰 + 예약분이 8K에 근접하면 분 리셋까지 대기 → "Request too large" 사전 차단
  • TPD 추적: Groq 일간 200K 토큰을 실시간 누적, 소진 시 남은 AI 호출을 SKIP하고 다음 실행에서 재처리
  • RPD 추적: Gemma는 TPM이 무제한이라 일간 요청 수(1,500)가 실질 상한 → 요청 단위로 누적·경고
  • 단일 워커: 병렬 처리는 8K TPM을 동시에 소진해 오히려 429를 유발 → 무료 티어에선 워커 1로 고정

부하·노이즈 최소화

  • 번역은 전체, 재시도는 고위험만: 모든 신규 CVE를 번역해 대시보드를 한글화하되, 무료 Gemma의 일시 오류(503 high demand / 500)에 대한 백오프 재시도는 고위험/에스컬레이션에만 적용 → 중요한 CVE의 한글화는 보장하고 저위험은 즉시 영문 폴백해 부하를 억제
  • 알림·룰은 고위험만: 무거운 Groq 룰 생성과 Slack/Issue는 고위험에만 → 희소 자원인 Groq TPD 보존
  • DB 용량 최소화: 상세 리포트(AI 분석 전문·룰 원문)는 GitHub Issue에 저장, DB에는 요약+링크만 → Supabase free tier(500MB) 방어

Thread-Safe로 API별 속도 + 분/일 토큰을 관리합니다.

API한도윈도우최소 간격

토큰 관리대상한도동작
TPM (분당 토큰)Groq8,000예약분이 한도 근접 시 분 리셋까지 대기
TPD (일간 토큰)Groq200,000소진 시 AI 호출 SKIP → 다음 실행 재처리
RPD (일간 요청)Gemma1,500요청 단위 누적, 90% 경고
  • 사용률 80%↑ 속도 조절, 90%↑ 추가 대기
  • 429 시 Retry-After 파싱 자동 대기 (분+초 복합: 10m3.072s → 603초)
  • 실행 종료 시 API·TPM·TPD·RPD 사용량 요약 출력

Usage

GitHub Actions: cron 자동 실행 (수동: Actions 탭 > Argus CVE Monitor > Run workflow)

로컬 실행:

export GH_TOKEN="..." SUPABASE_URL="..." SUPABASE_KEY="..."
export SLACK_WEBHOOK_URL="..." GROQ_API_KEY="..." GEMINI_API_KEY="..."

python src/main.py                    # CVE 파이프라인
python src/export_dashboard_data.py   # 대시보드 데이터 Export

Supabase 자격증명이 없으면 빈 샘플 데이터가 생성되어 대시보드가 에러 없이 로드됩니다.

Pipeline Details

1. Health Check └─ 환경 변수 검증(6개 필수) · assets.json 로드 2. 공식 룰 재탐색 ├─ AI 룰만 있는 CVE → 공식 룰 교체 / 룰 없는 고위험 → 새 공식 룰 └─ 배치 10건, 쿨다운(성공 7일 / 실패 1일) 3. 워터마크 수집 (누락 0) ├─ read_watermark() → 마지막 처리 지점 로드 (없으면 24h 부트스트랩) ├─ CISA KEV + VulnCheck KEV ├─ 워터마크 이후 커밋 전체 조회 → CVE별 최신 커밋시각 ├─ content_hash로 이미 처리분(is_new=False) 제거 └─ EPSS 배치 조회 4. FIFO 정렬 + 상한 └─ 커밋 오래된 순, 신규만 처리, 실행당 50건 (초과분 이월) 5. CVE 처리 (단일 워커) ── 3단계 라우팅 ├─ 위협 인텔리전스 보강 (NVD·SSVC·PoC·ExploitDB·MSF) ├─ 알림 판정 ├─ [T1 고위험/에스컬레이션] 번역(재시도) + Slack + Issue + AI 룰 │ ├─ 룰 생성 게이트(KEV/SSVC/MSF/ExploitDB/EPSS/PoC) │ ├─ AV 파싱 → 엔진 선택 (Sigma 필수 · AV:N→네트워크 · AV:L→YARA) │ └─ 파서/컴파일 자기검증 → trust 등급 부여 ├─ [T2 신규 저위험] 번역 + DB 추적 (알림·룰 없음) └─ [T3 기존·무변화] content_hash만 갱신 (Groq TPD 소진 시 AI 단계 SKIP → 다음 실행 재처리) 6. 워터마크 전진 └─ 미처리(실패+이월) 최소 커밋시각 앞까지만 전진 → 저장 7. Slack 배치 요약 + Rate Limit 요약(RPM·TPM·TPD·RPD)

Supabase Schema

-- CVE 이력 (상세 리포트는 GitHub Issue에, DB는 요약+링크만) CREATE TABLE cves ( id TEXT PRIMARY KEY, cvss_score REAL, epss_score REAL, is_kev BOOLEAN, has_official_rules BOOLEAN DEFAULT FALSE, last_alert_at TIMESTAMPTZ, last_alert_state JSONB, rules_snapshot JSONB, report_url TEXT, last_rule_check_at TIMESTAMPTZ, content_hash TEXT, updated_at TIMESTAMPTZ );

License & Compliance

개인이 개발했지만 회사 자산 점검에 활용하고 결과를 공개 대시보드/Issue에 재게시하므로, 데이터 소스별 라이선스와 개인정보 취급을 설계 단계부터 준수합니다.

준수 원칙

① 룰 재게시 시 출처·author·라이선스 고지 보존. 공개 룰을 재게시할 때 원 저작자 표기를 유지합니다 (예: SigmaHQ DRL 1.1은 author 표기 의무 — 룰 YAML의 author 필드 보존으로 충족).

② PoC 원문·기관 공지 원문은 재게시 금지, 링크만. Exploit-DB PoC 코드와 KrCERT/KISA 공지 원문은 대시보드/Issue에 재게시하지 않고 링크·요약만 제공합니다. AI 프롬프트의 내부 컨텍스트로만 활용합니다.

③ 무료 AI에 회사 내부 자산 정보 미전송. 무료 티어(Gemma/Groq)에는 공개된 CVE 데이터만 전송합니다. 무료 티어 입출력은 서비스 제공자의 제품 개선에 사용될 수 있으므로, 회사 내부 자산·인프라 정보는 프롬프트에 절대 넣지 않습니다.
